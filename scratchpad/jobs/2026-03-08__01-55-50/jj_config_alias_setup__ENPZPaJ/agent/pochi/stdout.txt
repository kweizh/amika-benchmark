The value is a\r\n[string pattern](./revsets.md#string-patterns) that matches the names of the\r\nbookmarks to track. This setting is applied to all new bookmarks, including ones\r\ncreated locally and ones fetched from the remote. For example:\r\n\r\n```toml\r\n[remotes.origin]\r\nauto-track-bookmarks = \"*\"\r\n```\r\n\r\nThis will simply track all bookmarks for the remote \"origin\". There are various\r\nreasons to restrict which bookmarks to track:\r\n\r\n- When collaborating with other people via the same remote, you may not want to\r\n  track all the bookmarks of your collaborators. Similarly, you may not want to\r\n  push some of your bookmarks to the remote at all; these may only be intended\r\n  for local use. Many people use a \"personal prefix\" in their bookmark names.\r\n  This marks a bookmark as belonging to one person, making it possible to track\r\n  and push only bookmarks with that prefix. For example, Alice may set her\r\n  configuration like so:\r\n\r\n  ```toml\r\n  [remotes.origin]\r\n  auto-track-bookmarks = \"alice/*\"\r\n  ```\r\n\r\n  That way, bookmarks pushed by other people (who probably use a different\r\n  prefix or none at all) are not tracked automatically. At the same time, Alice\r\n  can create bookmarks without the prefix for local-only use. You can also\r\n  configure Jujutsu to use your prefix for generated bookmark names; see the\r\n  section\r\n  [\"Generated bookmark names on push\"](#generated-bookmark-names-on-push).\r\n\r\n- Another reason to restrict the bookmarks to track may be that you're not\r\n  collaborating with people using the same remote, but a different one. For\r\n  example, if you make a fork on GitHub, your fork will usually be called\r\n  \"origin\", while the repository you forked from is usually called \"upstream\".\r\n  In that case, you may want to track all bookmarks from your fork, but only the\r\n  \"main\" bookmark from upstream. Here's an example how to achieve that:\r\n\r\n  ```toml\r\n  [remotes.origin]\r\n  auto-track-bookmarks = \"*\"\r\n  [remotes.upstream]\r\n  auto-track-bookmarks = \"main\"\r\n  ```\r\n\r\n- If you're collaborating with people on the same remote, but your bookmark\r\n  names don't follow a pattern that allows a bookmark's \"owner\" to be identified\r\n  (e.g. `alice/*`), you can use `auto-track-created-bookmarks` instead:\r\n\r\n  ```toml\r\n  [remotes.origin]\r\n  auto-track-created-bookmarks = \"*\"\r\n  ```\r\n\r\n  This pattern will only apply to bookmarks you create with `jj bookmark create`\r\n  and `jj bookmark set`. The disadvantage of this option is the fact that you\r\n  will have to track your own bookmarks manually if you fetch them from the\r\n  remote. This may happen regularly if you use multiple computers to work on the\r\n  same project.\r\n\r\n- Lastly, you may be working on various projects with different conventions for\r\n  bookmark names. In that case, it can be handy to apply different configuration\r\n  to different (groups of) repositories. Read about how to do that in the\r\n  section [\"Conditional variables\"](#conditional-variables).\r\n\r\n### Automatic local bookmark creation on `jj git clone`\r\n\r\nWhen cloning a new Git repository, `jj` by default creates a local bookmark\r\ntracking the default remote bookmark (such as `main` for `main@origin`.) If you\r\naren't going to update the `main` bookmark locally, the tracking bookmark isn't\r\nnecessary.\r\n\r\nThis behavior can be disabled by\r\n\r\n```toml\r\n[git]\r\ntrack-default-bookmark-on-clone = false\r\n```\r\n\r\n### Abandon commits that became unreachable in Git\r\n\r\nBy default, when `jj` imports refs from Git, it will look for commits that used\r\nto be [reachable][reachable] but no longer are reachable. Those commits will\r\nthen be abandoned, and any descendant commits will be rebased off of them (as\r\nusual when commits are abandoned). You can disable this behavior and instead\r\nleave the Git-unreachable commits in your repo by setting:\r\n\r\n```toml\r\n[git]\r\nabandon-unreachable-commits = false\r\n```\r\n\r\n[reachable]: https://git-scm.com/docs/gitglossary/#Documentation/gitglossary.txt-aiddefreachableareachable\r\n\r\n### Generated bookmark names on push\r\n\r\n`jj git push --change` generates bookmark names with a prefix of \"push-\" by\r\ndefault. You can pick a different prefix and formatting by setting the\r\n`templates.git_push_bookmark` template. For example:\r\n\r\n```toml\r\n[templates]\r\ngit_push_bookmark = '\"martinvonz/push-\" ++ change_id.short()'\r\n```\r\n\r\nThis template should include expressions like `change_id` to generate unique and\r\nstable bookmark.\r\n\r\n### Set of private commits\r\n\r\nYou can configure the set of private commits by setting `git.private-commits` to\r\na revset. The value is a revset of commits that Jujutsu will refuse to push. If\r\nunset, all commits are eligible to be pushed.\r\n\r\n```toml\r\n[git]\r\n# Prevent pushing work in progress or anything explicitly labeled \"private\"\r\nprivate-commits = \"description('wip:*') | description('private:*')\"\r\n```\r\n\r\nIf a commit is in `git.private-commits` but is already on the remote, then it is\r\nnot considered a private commit. Commits that are immutable are also excluded\r\nfrom the private set.\r\n\r\nPrivate commits prevent their descendants from being pushed, since doing so\r\nwould require pushing the private commit as well.\r\n\r\n### Git subprocessing behavior\r\n\r\nGit remote interactions are handled by spawning a `git` subprocess.\r\nIf `git` is not on your OS path, or you want to specify a particular binary,\r\nyou can:\r\n\r\n```toml\r\n[git]\r\nexecutable-path = \"/path/to/git\"\r\n```\r\n\r\n## Merge settings\r\n\r\n### Granularity of hunks\r\n\r\n`jj` by default resolves content conflicts by splitting text into line-level\r\nhunks and merge them. This can be configured to split hunks further into\r\nword-level hunks.\r\n\r\n* `line`: split into line hunks (default)\r\n* `word`: split into word hunks\r\n\r\n```toml\r\n[merge]\r\nhunk-level = \"line\"\r\n```\r\n\r\n### Resolution of same-change conflicts\r\n\r\n`jj` by default resolves conflicts if all sides made the same change. This\r\nmatches what Git and Mercurial do (in the 3-way case at least), but not what\r\nDarcs does. It also means that repeated 3-way merging of multiple trees may give\r\ndifferent results depending on the order of merging. To turn it off, set\r\n`same-change = \"keep\"`.\r\n\r\n* `keep`: leave same-change conflict unresolved\r\n* `accept`: resolve same-change conflict to new value (default)\r\n\r\n```toml\r\n[merge]\r\nsame-change = \"accept\"\r\n```\r\n\r\n## Filesystem monitor\r\n\r\nIn large repositories, it may be beneficial to use a \"filesystem monitor\" to\r\ntrack changes to the working copy. This allows `jj` to take working copy\r\nsnapshots without having to rescan the entire working copy.\r\n\r\nThis is governed by the `fsmonitor.backend` option. Currently, the valid values\r\nare `\"none\"` or `\"watchman\"`.\r\n\r\n### Watchman\r\n\r\nTo configure the Watchman filesystem monitor, set\r\n`fsmonitor.backend = \"watchman\"`. Ensure that you have [installed the Watchman\r\nexecutable on your system](https://facebook.github.io/watchman/docs/install).\r\n\r\nYou can configure `jj` to use watchman triggers to automatically create\r\nsnapshots on filesystem changes by setting\r\n`fsmonitor.watchman.register-snapshot-trigger = true`.\r\n\r\nYou can check whether Watchman is enabled and whether it is installed correctly\r\nusing `jj debug watchman status`.\r\n\r\nNote: `watchman` heavily uses `inotify` and sets up a user watch per-file. On\r\nlarge repositories, this may cause `watchman` to fail and commands like\r\n`jj status` to take longer than expected. If you experience this run\r\n`jj debug watchman status` and tune your `inotify` limits.\r\n\r\n## Snapshot settings\r\n\r\n### Paths to automatically track\r\n\r\nAll new files in the working copy that don't match the ignore patterns are\r\ntracked by default. You can set the `snapshot.auto-track` to set which paths\r\nget automatically tracked when they're added to the working copy. See the\r\n[fileset documentation](filesets.md) for the syntax. Files with paths matching\r\n[ignore files](working-copy.md#ignored-files) are never tracked automatically.\r\n\r\nIf you set `snapshot.auto-track` to a non-default value, untracked files can be\r\ntracked with `jj file track`.\r\n\r\nYou can use `jj file untrack` to untrack a file while keeping it in the working\r\ncopy. However, first [ignore](working-copy.md#ignored-files) them or remove them\r\nfrom the `snapshot.auto-track` patterns; otherwise they will be immediately\r\ntracked again.\r\n\r\n### Maximum size for new files\r\n\r\nBy default, as an anti-footgun measure, `jj` will refuse to add new files to the\r\nsnapshot that are larger than a certain size; the default is 1MiB. This can be\r\nchanged by setting `snapshot.max-new-file-size` to a different value. For\r\nexample:\r\n\r\n```toml\r\n[snapshot]\r\nmax-new-file-size = \"10MiB\"\r\n# the following is equivalent\r\nmax-new-file-size = 10485760\r\n```\r\n\r\nThe value can be specified using a human readable string with typical suffixes;\r\n`B`, `MiB`, `GB`, etc. By default, if no suffix is provided, or the value is a\r\nraw integer literal, the value is interpreted as if it were specified in bytes.\r\n\r\nFiles that already exist in the working copy are not subject to this limit.\r\n\r\nSetting this value to zero will disable the limit entirely.\r\n\r\n### Automatic update of stale working copies\r\n\r\nWhen a working copy becomes stale (meaning the working copy's recorded commit is\r\nno longer the current commit for that workspace), `jj` will normally prompt you\r\nto update it with `jj workspace update-stale`. You can configure `jj` to\r\nautomatically update stale working copies by setting:\r\n\r\n```toml\r\n[snapshot]\r\nauto-update-stale = true\r\n```\r\n\r\nThis is particularly useful if you have [multiple workspaces](working-copy.md#workspaces)\r\nand want to avoid manually updating each one.\r\n\r\nFor more information on stale working copies, see the [working copy\r\ndocumentation](working-copy.md#stale-working-copy).\r\n\r\n## Working copy settings\r\n\r\n### EOL conversion settings\r\n\r\nThis settings serves the same purpose as the [`core.autocrlf`][git-autocrlf] git\r\nconfig.\r\n\r\nThe line endings conversion won't be applied to files detected as binary files\r\nvia a heuristics[^1] regardless of the settings. This is similar to git.\r\n\r\n```toml\r\n[working-copy]\r\n# No EOL conversion. Similar to core.autocrlf = false.\r\neol-conversion = \"none\"\r\n# Apply CRLF to LF EOL conversion when we check files in the backend store from\r\n# the local file system but not apply EOL conversion when we check out the code\r\n# from the backend store to the local file system. Similar to core.autocrlf =\r\n# input.\r\neol-conversion = \"input\"\r\n# Setting this to \"input-output\" if you want to have CRLF line endings in your\r\n# working directory and the repository has LF line endings. Similar to\r\n# core.autocrlf = true.\r\neol-conversion = \"input-output\"\r\n```\r\n\r\n[git-autocrlf]: https://git-scm.com/book/en/v2/Customizing-Git-Git-Configuration#_core_autocrlf\r\n[gitoxide-is-binary]: https://github.com/GitoxideLabs/gitoxide/blob/073487b38ed40bcd7eb45dc110ae1ce84f9275a9/gix-filter/src/eol/utils.rs#L98-L100\r\n[git-is-binary]: https://github.com/git/git/blob/f1ca98f609f9a730b9accf24e5558a10a0b41b6c/convert.c#L94-L103\r\n[^1]: To detect if a file is binary, Jujutsu currently checks if there is NULL\r\n      byte in the file which is different from the algorithm of\r\n      [`gitoxide`][gitoxide-is-binary] or [`git`][git-is-binary]. Jujutsu\r\n      doesn't plan to align the binary detection logic with git.\r\n\r\n### Respect or ignore executable bit permission changes\r\n\r\nWhether to respect or ignore changes to the executable bit for files on Unix.\r\nThis option is unused on Windows.\r\n\r\n```toml\r\n[working-copy]\r\nexec-bit-change = \"respect\" | \"ignore\" | \"auto\" (default)\r\n```\r\n\r\nOn Unix systems, files have a permission bit for whether they are executable as\r\nscripts or binary code. Jujutsu stores this state in the repository and will\r\nupdate it for files as you operate on a repository. If you set your working\r\ncopy to a commit where a file is recorded as executable or not, `jj` will\r\nadjust the permission of that file. If you change a file's executable bit\r\nthrough the filesystem, `jj` will record that change when taking a snapshot.\r\n\r\nSetting this to `\"ignore\"` will make Jujutsu ignore the executable bit on the\r\nfilesystem when updating the state for the repository. In addition, `jj` will\r\nnot attempt to modify a file's executable bit and will add new files as\r\n\"not executable.\" This is already the behavior on Windows, and having the\r\noption to enable this behavior is useful for Unix users dual-booting Windows,\r\nWindows users accessing files from WSL, or anyone experimenting with other\r\nfilesystem configurations.\r\n\r\nOn Unix if `\"auto\"` is set (the default), `jj` will try to detect whether the\r\nfilesystem supports changing executable bits to choose between `\"respect\"` or\r\n`\"ignore\"`. On errors, we assume `\"respect\"`, except if permission was denied\r\nwhich will assume `\"ignore\"`.\r\n\r\nOn Windows, files have no executable bit so this option is unused.\r\n\r\nYou can always use `jj file chmod` to update the recorded executable bit for a\r\nfile manually. If this option is `\"respect\"`, `jj` will also attempt to\r\npropagate that change to the filesystem.\r\n\r\nNote that if you modify a file's executable bit before changing this setting\r\nfrom `\"ignore\"` or  `\"auto\"` to `\"respect\"`, `jj` may not update the stored\r\nexecutable bit until you modify the file's contents or update its modification\r\ntime, e.g. with `touch`.\r\n\r\n## Ways to specify `jj` config: details\r\n\r\n### User config files\r\n\r\nAn easy way to find the user config file/directory is:\r\n\r\n```bash\r\njj config path --user\r\n```\r\n\r\nOn all platforms, the user's global `jj` configurations are by default loaded in\r\nthe following precedence order (with later configs overriding earlier ones):\r\n\r\n- `$HOME/.jjconfig.toml`\r\n- `<PLATFORM_SPECIFIC>/jj/config.toml` (preferred)\r\n- `<PLATFORM_SPECIFIC>/jj/conf.d/*.toml`\r\n\r\nwhere `$HOME` represents the user's home directory (`%USERPROFILE%` on Windows),\r\nand `<PLATFORM_SPECIFIC>` represents the platform-specific configuration\r\ndirectory shown in the table below. The platform-specific location is\r\nrecommended for better integration with platform services.\r\n\r\nThe files in the `conf.d` directory are loaded in lexicographic order. This\r\nallows configs to be split across multiple files and combines well with\r\n[Conditional Variables](#conditional-variables).\r\n\r\n| Platform        | Location of `<PLATFORM_SPECIFIC>` dir | Example config file location                              |\r\n| :-------------- | :------------------------------------ | :-------------------------------------------------------- |\r\n| Linux and macOS | `$XDG_CONFIG_HOME` or `$HOME/.config` | `/home/alice/.config/jj/config.toml`                      |\r\n| Windows         | `{FOLDERID_RoamingAppData}`           | `C:\\Users\\Alice\\AppData\\Roaming\\jj\\config.toml`           |\r\n\r\nThe location of the `jj` user config files/directories can also be overridden\r\nwith the `JJ_CONFIG` environment variable. If the environment variable is set\r\n(even to an empty string), it will be used instead of any configuration files in\r\nthe default locations. It can be a path to a TOML file or a directory of TOML\r\nfiles, which will be loaded in lexicographic order and merged. Multiple paths\r\ncan be specified by separating them with a platform-specific path separator (`:`\r\non Unix-like systems, `;` on Windows).\r\n\r\nFor example, the following could be used to run `jj` without loading any user\r\nconfigs:\r\n\r\n```bash\r\nJJ_CONFIG= jj log       # Ignores any settings specified in any config files.\r\n```\r\n\r\nThere are also the `--config-file <PATH>` and `--config <NAME=VALUE>`\r\n[global options](./cli-reference.md#options) which work with any `jj` command.\r\n\r\n### JSON Schema Support\r\n\r\nMany popular editors support TOML file syntax highlighting and validation. To\r\nenable schema validation in your editor, add this line at the top of your TOML\r\nconfig files:\r\n\r\n```toml\r\n#:schema https://docs.jj-vcs.dev/latest/config-schema.json\r\n```\r\n\r\nThis enables features like:\r\n\r\n- Autocomplete for config keys\r\n- Type checking of values\r\n- Documentation on hover\r\n- Validation of settings\r\n\r\nHere are some popular editors with TOML schema validation support:\r\n\r\n- VS Code\r\n  - Install [Even Better TOML](https://marketplace.visualstudio.com/items?itemName=tamasfe.even-better-toml)\r\n\r\n- Neovim/Vim\r\n  - Use with [nvim-lspconfig](https://github.com/neovim/nvim-lspconfig) and [taplo](https://github.com/tamasfe/taplo)\r\n\r\n- Helix\r\n  - Install [taplo](https://github.com/tamasfe/taplo)\r\n\r\n- JetBrains IDEs (IntelliJ, PyCharm, etc)\r\n  - Install [TOML](https://plugins.jetbrains.com/plugin/8195-toml) plugin\r\n\r\n- Emacs\r\n  - Install [lsp-mode](https://github.com/emacs-lsp/lsp-mode) and [toml-mode](https://github.com/dryman/toml-mode.el)\r\n  - Configure [taplo](https://github.com/tamasfe/taplo) as the LSP server\r\n\r\n### Specifying config on the command-line\r\n\r\nYou can use one or more `--config`/`--config-file` options on the command line\r\nto specify additional configuration settings. This overrides settings defined in\r\nconfig files or environment variables. For example,\r\n\r\n```shell\r\n# Must not have spaces around the `=`\r\njj --config ui.color=always --config ui.diff-editor=meld split\r\n```\r\n\r\nConfig value should be specified as a TOML expression.
