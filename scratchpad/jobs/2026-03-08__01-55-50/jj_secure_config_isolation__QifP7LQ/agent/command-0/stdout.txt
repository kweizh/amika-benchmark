This might be preferred if the\r\nsigning backend requires user interaction or is slow, so that signing is\r\nperformed in a single batch operation.\r\n\r\n```toml\r\n# Configure signing backend as before, but lazily signing only on push.\r\n[signing]\r\nbehavior = \"drop\"\r\nbackend = \"ssh\"\r\nkey = \"ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGj+J6N6SO+4P8dOZqfR1oiay2yxhhHnagH52avUqw5h\"\r\n\r\n[git]\r\nsign-on-push = true\r\n```\r\n\r\n## Commit Signature Verification\r\n\r\nBy default signature verification and display is **disabled** as it incurs a\r\nperformance cost when rendering medium to large change logs. You can enable it\r\nby setting `ui.show-cryptographic-signatures` to true in your configuration.\r\n\r\nIf you want to display commit signatures in your templates, you can use\r\n`commit.signature()` (see [Commit type](./templates.md#commit-type)). The\r\nreturned [CryptographicSignature\r\nType](./templates.md#cryptographicsignature-type) provides methods to retrieve\r\nsignature details.\r\n\r\n## Git settings\r\n\r\n### Default colocation\r\n\r\nWhen creating a Git-backed Jujutsu repository, they use \"colocated\" mode by\r\ndefault, which places the `.git` directory next to the `.jj` directory.\r\nColocation allows some amount of two-way interoperability, but it can perform\r\nworse in large repos.\r\n\r\nThe setting `git.colocate` is a boolean option that controls whether or not the\r\n`jj git init` and `jj git clone` commands should create colocated workspace by\r\ndefault. Set `git.colocate` to `false` to disable it.\r\n\r\nSee [Colocated Jujutsu/Git workspaces](git-compatibility.md#colocated-jujutsugit-repos)\r\nfor more information.\r\n\r\n### Default remotes for `jj git fetch` and `jj git push`\r\n\r\nBy default, if a single remote exists it is used for `jj git fetch` and `jj git\r\npush`; however if multiple remotes exist, the default remote is assumed to be\r\nnamed `\"origin\"`, just like in Git. Sometimes this is undesirable, e.g. when you\r\nwant to fetch from a different remote than you push to, such as a GitHub fork.\r\n\r\nTo change this behavior, you can modify the [repository\r\nconfiguration](#config-files-and-toml) variable `git.fetch`, which can be a\r\nsingle remote, or a list of remotes to fetch from multiple places:\r\n\r\n```sh\r\njj config set --repo git.fetch \"upstream\"\r\njj config set --repo git.fetch '[\"origin\", \"upstream\"]'\r\n```\r\n\r\nBy default, the specified pattern matches remote names with glob syntax. You can\r\nalso use other [string pattern syntax](revsets.md#string-patterns):\r\n\r\n```sh\r\njj config set --repo git.fetch \"regex:'^(remote|upstream)'\"\r\njj config set --repo git.fetch '[\"remote*\", \"upstream*\"]'\r\n```\r\n\r\nSimilarly, you can also set the variable `git.push` to cause `jj git push` to\r\npush to a different remote:\r\n\r\n```sh\r\njj config set --repo git.push \"github\"\r\n```\r\n\r\nNote that unlike `git.fetch`, `git.push` can currently only be a single remote.\r\nThis is not a hard limitation, and could be changed in the future if there is\r\ndemand.\r\n\r\n### Automatic tracking of bookmarks\r\n\r\nYou can configure which bookmarks to track automatically per remote, using the\r\n`remotes.<name>.auto-track-bookmarks` config. The value is a\r\n[string pattern](./revsets.md#string-patterns) that matches the names of the\r\nbookmarks to track. This setting is applied to all new bookmarks, including ones\r\ncreated locally and ones fetched from the remote. For example:\r\n\r\n```toml\r\n[remotes.origin]\r\nauto-track-bookmarks = \"*\"\r\n```\r\n\r\nThis will simply track all bookmarks for the remote \"origin\". There are various\r\nreasons to restrict which bookmarks to track:\r\n\r\n- When collaborating with other people via the same remote, you may not want to\r\n  track all the bookmarks of your collaborators. Similarly, you may not want to\r\n  push some of your bookmarks to the remote at all; these may only be intended\r\n  for local use. Many people use a \"personal prefix\" in their bookmark names.\r\n  This marks a bookmark as belonging to one person, making it possible to track\r\n  and push only bookmarks with that prefix. For example, Alice may set her\r\n  configuration like so:\r\n\r\n  ```toml\r\n  [remotes.origin]\r\n  auto-track-bookmarks = \"alice/*\"\r\n  ```\r\n\r\n  That way, bookmarks pushed by other people (who probably use a different\r\n  prefix or none at all) are not tracked automatically. At the same time, Alice\r\n  can create bookmarks without the prefix for local-only use. You can also\r\n  configure Jujutsu to use your prefix for generated bookmark names; see the\r\n  section\r\n  [\"Generated bookmark names on push\"](#generated-bookmark-names-on-push).\r\n\r\n- Another reason to restrict the bookmarks to track may be that you're not\r\n  collaborating with people using the same remote, but a different one. For\r\n  example, if you make a fork on GitHub, your fork will usually be called\r\n  \"origin\", while the repository you forked from is usually called \"upstream\".\r\n  In that case, you may want to track all bookmarks from your fork, but only the\r\n  \"main\" bookmark from upstream. Here's an example how to achieve that:\r\n\r\n  ```toml\r\n  [remotes.origin]\r\n  auto-track-bookmarks = \"*\"\r\n  [remotes.upstream]\r\n  auto-track-bookmarks = \"main\"\r\n  ```\r\n\r\n- If you're collaborating with people on the same remote, but your bookmark\r\n  names don't follow a pattern that allows a bookmark's \"owner\" to be identified\r\n  (e.g. `alice/*`), you can use `auto-track-created-bookmarks` instead:\r\n\r\n  ```toml\r\n  [remotes.origin]\r\n  auto-track-created-bookmarks = \"*\"\r\n  ```\r\n\r\n  This pattern will only apply to bookmarks you create with `jj bookmark create`\r\n  and `jj bookmark set`. The disadvantage of this option is the fact that you\r\n  will have to track your own bookmarks manually if you fetch them from the\r\n  remote. This may happen regularly if you use multiple computers to work on the\r\n  same project.\r\n\r\n- Lastly, you may be working on various projects with different conventions for\r\n  bookmark names. In that case, it can be handy to apply different configuration\r\n  to different (groups of) repositories. Read about how to do that in the\r\n  section [\"Conditional variables\"](#conditional-variables).\r\n\r\n### Automatic local bookmark creation on `jj git clone`\r\n\r\nWhen cloning a new Git repository, `jj` by default creates a local bookmark\r\ntracking the default remote bookmark (such as `main` for `main@origin`.) If you\r\naren't going to update the `main` bookmark locally, the tracking bookmark isn't\r\nnecessary.\r\n\r\nThis behavior can be disabled by\r\n\r\n```toml\r\n[git]\r\ntrack-default-bookmark-on-clone = false\r\n```\r\n\r\n### Abandon commits that became unreachable in Git\r\n\r\nBy default, when `jj` imports refs from Git, it will look for commits that used\r\nto be [reachable][reachable] but no longer are reachable. Those commits will\r\nthen be abandoned, and any descendant commits will be rebased off of them (as\r\nusual when commits are abandoned). You can disable this behavior and instead\r\nleave the Git-unreachable commits in your repo by setting:\r\n\r\n```toml\r\n[git]\r\nabandon-unreachable-commits = false\r\n```\r\n\r\n[reachable]: https://git-scm.com/docs/gitglossary/#Documentation/gitglossary.txt-aiddefreachableareachable\r\n\r\n### Generated bookmark names on push\r\n\r\n`jj git push --change` generates bookmark names with a prefix of \"push-\" by\r\ndefault. You can pick a different prefix and formatting by setting the\r\n`templates.git_push_bookmark` template. For example:\r\n\r\n```toml\r\n[templates]\r\ngit_push_bookmark = '\"martinvonz/push-\" ++ change_id.short()'\r\n```\r\n\r\nThis template should include expressions like `change_id` to generate unique and\r\nstable bookmark.\r\n\r\n### Set of private commits\r\n\r\nYou can configure the set of private commits by setting `git.private-commits` to\r\na revset. The value is a revset of commits that Jujutsu will refuse to push. If\r\nunset, all commits are eligible to be pushed.\r\n\r\n```toml\r\n[git]\r\n# Prevent pushing work in progress or anything explicitly labeled \"private\"\r\nprivate-commits = \"description('wip:*') | description('private:*')\"\r\n```\r\n\r\nIf a commit is in `git.private-commits` but is already on the remote, then it is\r\nnot considered a private commit. Commits that are immutable are also excluded\r\nfrom the private set.\r\n\r\nPrivate commits prevent their descendants from being pushed, since doing so\r\nwould require pushing the private commit as well.\r\n\r\n### Git subprocessing behavior\r\n\r\nGit remote interactions are handled by spawning a `git` subprocess.\r\nIf `git` is not on your OS path, or you want to specify a particular binary,\r\nyou can:\r\n\r\n```toml\r\n[git]\r\nexecutable-path = \"/path/to/git\"\r\n```\r\n\r\n## Merge settings\r\n\r\n### Granularity of hunks\r\n\r\n`jj` by default resolves content conflicts by splitting text into line-level\r\nhunks and merge them. This can be configured to split hunks further into\r\nword-level hunks.\r\n\r\n* `line`: split into line hunks (default)\r\n* `word`: split into word hunks\r\n\r\n```toml\r\n[merge]\r\nhunk-level = \"line\"\r\n```\r\n\r\n### Resolution of same-change conflicts\r\n\r\n`jj` by default resolves conflicts if all sides made the same change. This\r\nmatches what Git and Mercurial do (in the 3-way case at least), but not what\r\nDarcs does. It also means that repeated 3-way merging of multiple trees may give\r\ndifferent results depending on the order of merging. To turn it off, set\r\n`same-change = \"keep\"`.\r\n\r\n* `keep`: leave same-change conflict unresolved\r\n* `accept`: resolve same-change conflict to new value (default)\r\n\r\n```toml\r\n[merge]\r\nsame-change = \"accept\"\r\n```\r\n\r\n## Filesystem monitor\r\n\r\nIn large repositories, it may be beneficial to use a \"filesystem monitor\" to\r\ntrack changes to the working copy. This allows `jj` to take working copy\r\nsnapshots without having to rescan the entire working copy.\r\n\r\nThis is governed by the `fsmonitor.backend` option. Currently, the valid values\r\nare `\"none\"` or `\"watchman\"`.\r\n\r\n### Watchman\r\n\r\nTo configure the Watchman filesystem monitor, set\r\n`fsmonitor.backend = \"watchman\"`. Ensure that you have [installed the Watchman\r\nexecutable on your system](https://facebook.github.io/watchman/docs/install).\r\n\r\nYou can configure `jj` to use watchman triggers to automatically create\r\nsnapshots on filesystem changes by setting\r\n`fsmonitor.watchman.register-snapshot-trigger = true`.\r\n\r\nYou can check whether Watchman is enabled and whether it is installed correctly\r\nusing `jj debug watchman status`.\r\n\r\nNote: `watchman` heavily uses `inotify` and sets up a user watch per-file. On\r\nlarge repositories, this may cause `watchman` to fail and commands like\r\n`jj status` to take longer than expected. If you experience this run\r\n`jj debug watchman status` and tune your `inotify` limits.\r\n\r\n## Snapshot settings\r\n\r\n### Paths to automatically track\r\n\r\nAll new files in the working copy that don't match the ignore patterns are\r\ntracked by default. You can set the `snapshot.auto-track` to set which paths\r\nget automatically tracked when they're added to the working copy. See the\r\n[fileset documentation](filesets.md) for the syntax. Files with paths matching\r\n[ignore files](working-copy.md#ignored-files) are never tracked automatically.\r\n\r\nIf you set `snapshot.auto-track` to a non-default value, untracked files can be\r\ntracked with `jj file track`.\r\n\r\nYou can use `jj file untrack` to untrack a file while keeping it in the working\r\ncopy. However, first [ignore](working-copy.md#ignored-files) them or remove them\r\nfrom the `snapshot.auto-track` patterns; otherwise they will be immediately\r\ntracked again.\r\n\r\n### Maximum size for new files\r\n\r\nBy default, as an anti-footgun measure, `jj` will refuse to add new files to the\r\nsnapshot that are larger than a certain size; the default is 1MiB. This can be\r\nchanged by setting `snapshot.max-new-file-size` to a different value. For\r\nexample:\r\n\r\n```toml\r\n[snapshot]\r\nmax-new-file-size = \"10MiB\"\r\n# the following is equivalent\r\nmax-new-file-size = 10485760\r\n```\r\n\r\nThe value can be specified using a human readable string with typical suffixes;\r\n`B`, `MiB`, `GB`, etc. By default, if no suffix is provided, or the value is a\r\nraw integer literal, the value is interpreted as if it were specified in bytes.\r\n\r\nFiles that already exist in the working copy are not subject to this limit.\r\n\r\nSetting this value to zero will disable the limit entirely.\r\n\r\n### Automatic update of stale working copies\r\n\r\nWhen a working copy becomes stale (meaning the working copy's recorded commit is\r\nno longer the current commit for that workspace), `jj` will normally prompt you\r\nto update it with `jj workspace update-stale`. You can configure `jj` to\r\nautomatically update stale working copies by setting:\r\n\r\n```toml\r\n[snapshot]\r\nauto-update-stale = true\r\n```\r\n\r\nThis is particularly useful if you have [multiple workspaces](working-copy.md#workspaces)\r\nand want to avoid manually updating each one.\r\n\r\nFor more information on stale working copies, see the [working copy\r\ndocumentation](working-copy.md#stale-working-copy).\r\n\r\n## Working copy settings\r\n\r\n### EOL conversion settings\r\n\r\nThis settings serves the same purpose as the [`core.autocrlf`][git-autocrlf] git\r\nconfig.\r\n\r\nThe line endings conversion won't be applied to files detected as binary files\r\nvia a heuristics[^1] regardless of the settings.
