# Recover Lost Work Using the Operation Log ## Scenario You are a support engineer who has been given access to a colleague's Jujutsu (`jj`) repository in `/home/user/project`. Your colleague reported that they were working on a feature in a branch called `feature`, but after running a few commands, the branch and its associated commit seem to have disappeared. They are unsure what went wrong. Your job is to investigate the operation log, identify the operation that caused the loss, and restore the repository to the state before the accidental operation occurred. ## Repository State The repository is located at `/home/user/project`. It is a standard `jj`-backed Git repository. At some point during your colleague's session, an operation was performed that deleted or lost the `feature` bookmark and the commit it pointed to. Before starting any recovery steps, take time to understand the current state of the repository: - Run `jj log` to see the current revision history. - Run `jj bookmark list` to check which bookmarks currently exist. - Note that the `feature` bookmark is **missing** from the current state. ## Your Task You must complete the following steps: ### Step 1: Inspect the Operation Log Use the appropriate `jj` command to display the operation log for the repository. The operation log records every mutating operation performed on the repository, in reverse chronological order (newest first). Look through the log carefully. Each entry will show: - A short operation ID - A timestamp - A description of what the operation did Find the operation entry that corresponds to the action that deleted or lost the `feature` bookmark or the commit it pointed to. The description in the operation log should give you a clue about what happened. ### Step 2: Identify the Problematic Operation Based on your inspection of the operation log, determine which operation caused the loss. You are looking for an operation whose description clearly indicates it abandoned, deleted, or lost the commit or bookmark named `feature`. ### Step 3: Undo the Problematic Operation Use `jj undo` to undo the **most recent** operation (which is the one that caused the loss). The `jj undo` command undoes the last mutating operation, restoring the repository to the state it was in just before that operation ran. After running `jj undo`, verify that the recovery was successful: - Run `jj log` and confirm that a commit with the description `"implement feature"` is now visible in the revision history. - Run `jj bookmark list` and confirm that the `feature` bookmark is now present and points to the expected commit. ### Step 4: Verify the Recovered State After the undo, the repository should satisfy all of the following conditions: 1. The `feature` bookmark exists and is present in the output of `jj bookmark list`. 2. The commit with the description `"implement feature"` exists and is reachable in `jj log`. 3. The `feature` bookmark points to the commit with the description `"implement feature"`. 4. The file `/home/user/project/feature.txt` exists and contains the text `feature work` (this file is part of the recovered commit). ## Constraints - Work only within `/home/user/project`. - Use only `jj` commands (no `git` commands, no manual file editing). - Do not use `sudo`. - Do not run `jj op restore` — use `jj undo` instead. ## Hints - The operation log is displayed with `jj op log` (or equivalently `jj operation log`). - `jj undo` undoes the most recent operation. Since the accidental operation is the most recent one in this scenario, a single `jj undo` is sufficient. - After undoing, you can inspect the result with `jj log` and `jj bookmark list` to confirm the state is correct. - Each operation in `jj op log` has an ID prefix that you can use with other operation-related commands if needed. ## Expected Outcome After completing the task: - `jj bookmark list` shows the `feature` bookmark. - `jj log` shows the commit with description `"implement feature"`. - The file `/home/user/project/feature.txt` exists with the content `feature work`.
