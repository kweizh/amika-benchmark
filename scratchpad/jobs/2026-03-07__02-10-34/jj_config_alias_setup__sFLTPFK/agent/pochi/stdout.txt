You can set the `snapshot.auto-track` to set which paths\r\nget automatically tracked when they're added to the working copy. See the\r\n[fileset documentation](filesets.md) for the syntax. Files with paths matching\r\n[ignore files](working-copy.md#ignored-files) are never tracked automatically.\r\n\r\nIf you set `snapshot.auto-track` to a non-default value, untracked files can be\r\ntracked with `jj file track`.\r\n\r\nYou can use `jj file untrack` to untrack a file while keeping it in the working\r\ncopy. However, first [ignore](working-copy.md#ignored-files) them or remove them\r\nfrom the `snapshot.auto-track` patterns; otherwise they will be immediately\r\ntracked again.\r\n\r\n### Maximum size for new files\r\n\r\nBy default, as an anti-footgun measure, `jj` will refuse to add new files to the\r\nsnapshot that are larger than a certain size; the default is 1MiB. This can be\r\nchanged by setting `snapshot.max-new-file-size` to a different value. For\r\nexample:\r\n\r\n```toml\r\n[snapshot]\r\nmax-new-file-size = \"10MiB\"\r\n# the following is equivalent\r\nmax-new-file-size = 10485760\r\n```\r\n\r\nThe value can be specified using a human readable string with typical suffixes;\r\n`B`, `MiB`, `GB`, etc. By default, if no suffix is provided, or the value is a\r\nraw integer literal, the value is interpreted as if it were specified in bytes.\r\n\r\nFiles that already exist in the working copy are not subject to this limit.\r\n\r\nSetting this value to zero will disable the limit entirely.\r\n\r\n### Automatic update of stale working copies\r\n\r\nWhen a working copy becomes stale (meaning the working copy's recorded commit is\r\nno longer the current commit for that workspace), `jj` will normally prompt you\r\nto update it with `jj workspace update-stale`. You can configure `jj` to\r\nautomatically update stale working copies by setting:\r\n\r\n```toml\r\n[snapshot]\r\nauto-update-stale = true\r\n```\r\n\r\nThis is particularly useful if you have [multiple workspaces](working-copy.md#workspaces)\r\nand want to avoid manually updating each one.\r\n\r\nFor more information on stale working copies, see the [working copy\r\ndocumentation](working-copy.md#stale-working-copy).\r\n\r\n## Working copy settings\r\n\r\n### EOL conversion settings\r\n\r\nThis settings serves the same purpose as the [`core.autocrlf`][git-autocrlf] git\r\nconfig.\r\n\r\nThe line endings conversion won't be applied to files detected as binary files\r\nvia a heuristics[^1] regardless of the settings. This is similar to git.\r\n\r\n```toml\r\n[working-copy]\r\n# No EOL conversion. Similar to core.autocrlf = false.\r\neol-conversion = \"none\"\r\n# Apply CRLF to LF EOL conversion when we check files in the backend store from\r\n# the local file system but not apply EOL conversion when we check out the code\r\n# from the backend store to the local file system. Similar to core.autocrlf =\r\n# input.\r\neol-conversion = \"input\"\r\n# Setting this to \"input-output\" if you want to have CRLF line endings in your\r\n# working directory and the repository has LF line endings. Similar to\r\n# core.autocrlf = true.\r\neol-conversion = \"input-output\"\r\n```\r\n\r\n[git-autocrlf]: https://git-scm.com/book/en/v2/Customizing-Git-Git-Configuration#_core_autocrlf\r\n[gitoxide-is-binary]: https://github.com/GitoxideLabs/gitoxide/blob/073487b38ed40bcd7eb45dc110ae1ce84f9275a9/gix-filter/src/eol/utils.rs#L98-L100\r\n[git-is-binary]: https://github.com/git/git/blob/f1ca98f609f9a730b9accf24e5558a10a0b41b6c/convert.c#L94-L103\r\n[^1]: To detect if a file is binary, Jujutsu currently checks if there is NULL\r\n      byte in the file which is different from the algorithm of\r\n      [`gitoxide`][gitoxide-is-binary] or [`git`][git-is-binary]. Jujutsu\r\n      doesn't plan to align the binary detection logic with git.\r\n\r\n### Respect or ignore executable bit permission changes\r\n\r\nWhether to respect or ignore changes to the executable bit for files on Unix.\r\nThis option is unused on Windows.\r\n\r\n```toml\r\n[working-copy]\r\nexec-bit-change = \"respect\" | \"ignore\" | \"auto\" (default)\r\n```\r\n\r\nOn Unix systems, files have a permission bit for whether they are executable as\r\nscripts or binary code. Jujutsu stores this state in the repository and will\r\nupdate it for files as you operate on a repository. If you set your working\r\ncopy to a commit where a file is recorded as executable or not, `jj` will\r\nadjust the permission of that file. If you change a file's executable bit\r\nthrough the filesystem, `jj` will record that change when taking a snapshot.\r\n\r\nSetting this to `\"ignore\"` will make Jujutsu ignore the executable bit on the\r\nfilesystem when updating the state for the repository. In addition, `jj` will\r\nnot attempt to modify a file's executable bit and will add new files as\r\n\"not executable.\" This is already the behavior on Windows, and having the\r\noption to enable this behavior is useful for Unix users dual-booting Windows,\r\nWindows users accessing files from WSL, or anyone experimenting with other\r\nfilesystem configurations.\r\n\r\nOn Unix if `\"auto\"` is set (the default), `jj` will try to detect whether the\r\nfilesystem supports changing executable bits to choose between `\"respect\"` or\r\n`\"ignore\"`. On errors, we assume `\"respect\"`, except if permission was denied\r\nwhich will assume `\"ignore\"`.\r\n\r\nOn Windows, files have no executable bit so this option is unused.\r\n\r\nYou can always use `jj file chmod` to update the recorded executable bit for a\r\nfile manually. If this option is `\"respect\"`, `jj` will also attempt to\r\npropagate that change to the filesystem.\r\n\r\nNote that if you modify a file's executable bit before changing this setting\r\nfrom `\"ignore\"` or  `\"auto\"` to `\"respect\"`, `jj` may not update the stored\r\nexecutable bit until you modify the file's contents or update its modification\r\ntime, e.g. with `touch`.\r\n\r\n## Ways to specify `jj` config: details\r\n\r\n### User config files\r\n\r\nAn easy way to find the user config file/directory is:\r\n\r\n```bash\r\njj config path --user\r\n```\r\n\r\nOn all platforms, the user's global `jj` configurations are by default loaded in\r\nthe following precedence order (with later configs overriding earlier ones):\r\n\r\n- `$HOME/.jjconfig.toml`\r\n- `<PLATFORM_SPECIFIC>/jj/config.toml` (preferred)\r\n- `<PLATFORM_SPECIFIC>/jj/conf.d/*.toml`\r\n\r\nwhere `$HOME` represents the user's home directory (`%USERPROFILE%` on Windows),\r\nand `<PLATFORM_SPECIFIC>` represents the platform-specific configuration\r\ndirectory shown in the table below. The platform-specific location is\r\nrecommended for better integration with platform services.\r\n\r\nThe files in the `conf.d` directory are loaded in lexicographic order. This\r\nallows configs to be split across multiple files and combines well with\r\n[Conditional Variables](#conditional-variables).\r\n\r\n| Platform        | Location of `<PLATFORM_SPECIFIC>` dir | Example config file location                              |\r\n| :-------------- | :------------------------------------ | :-------------------------------------------------------- |\r\n| Linux and macOS | `$XDG_CONFIG_HOME` or `$HOME/.config` | `/home/alice/.config/jj/config.toml`                      |\r\n| Windows         | `{FOLDERID_RoamingAppData}`           | `C:\\Users\\Alice\\AppData\\Roaming\\jj\\config.toml`           |\r\n\r\nThe location of the `jj` user config files/directories can also be overridden\r\nwith the `JJ_CONFIG` environment variable. If the environment variable is set\r\n(even to an empty string), it will be used instead of any configuration files in\r\nthe default locations. It can be a path to a TOML file or a directory of TOML\r\nfiles, which will be loaded in lexicographic order and merged. Multiple paths\r\ncan be specified by separating them with a platform-specific path separator (`:`\r\non Unix-like systems, `;` on Windows).\r\n\r\nFor example, the following could be used to run `jj` without loading any user\r\nconfigs:\r\n\r\n```bash\r\nJJ_CONFIG= jj log       # Ignores any settings specified in any config files.\r\n```\r\n\r\nThere are also the `--config-file <PATH>` and `--config <NAME=VALUE>`\r\n[global options](./cli-reference.md#options) which work with any `jj` command.\r\n\r\n### JSON Schema Support\r\n\r\nMany popular editors support TOML file syntax highlighting and validation. To\r\nenable schema validation in your editor, add this line at the top of your TOML\r\nconfig files:\r\n\r\n```toml\r\n#:schema https://docs.jj-vcs.dev/latest/config-schema.json\r\n`
