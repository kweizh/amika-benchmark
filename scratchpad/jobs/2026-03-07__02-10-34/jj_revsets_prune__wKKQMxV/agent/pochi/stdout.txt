revisions by using functions. Some functions take other\r\nrevsets (expressions) as arguments.\r\n\r\n??? note \"Function argument syntax\"\r\n\r\n    In this documentation, optional arguments are indicated with square\r\n    brackets like `[arg]`. Some arguments also have an optional label which can\r\n    be used to specify that argument without specifying all previous arguments.\r\n\r\n    For instance, `remote_bookmarks([name_pattern], [[remote=]remote_pattern])`\r\n    indicates that all of the following usages are valid:\r\n\r\n    * `remote_bookmarks()`\r\n    * `remote_bookmarks(\"main\")`\r\n    * `remote_bookmarks(\"main\", \"origin\")`\r\n    * `remote_bookmarks(\"main\", remote=\"origin\")`\r\n    * `remote_bookmarks(remote=\"origin\")`\r\n\r\n* `parents(x, [depth])`: `parents(x)` is the same as `x-`.\r\n  `parents(x, depth)` returns the parents of `x` at the given `depth`. For\r\n  instance, `parents(x, 3)` is equivalent to `x---`.\r\n\r\n* `children(x, [depth])`: `children(x)` is the same as `x+`.\r\n  `children(x, depth)` returns the children of `x` at the given `depth`. For\r\n  instance, `children(x, 3)` is equivalent to `x+++`.\r\n\r\n* `ancestors(x, [depth])`: `ancestors(x)` is the same as `::x`.\r\n  `ancestors(x, depth)` returns the ancestors of `x` limited to the given\r\n  `depth`.\r\n\r\n* `descendants(x, [depth])`: `descendants(x)` is the same as `x::`.\r\n  `descendants(x, depth)` returns the descendants of `x` limited to the given\r\n  `depth`.\r\n\r\n* `first_parent(x, [depth])`: `first_parent(x)` is similar to `parents(x)`, but\r\n  for merges, it only returns the first parent instead of returning all parents.\r\n  The `depth` argument also works similarly, so `first_parent(x, 2)` is\r\n  equivalent to `first_parent(first_parent(x))`.\r\n\r\n* `first_ancestors(x, [depth])`: Similar to `ancestors(x, [depth])`, but only\r\n  traverses the first parent of each commit. In Git, the first parent of a merge\r\n  commit is conventionally the branch into which changes are being merged, so\r\n  `first_ancestors()` can be used to exclude changes made on other branches.\r\n\r\n* `reachable(srcs, domain)`: All commits reachable from `srcs` within\r\n  `domain`, traversing all parent and child edges. `srcs` outside `domain` are\r\n  not considered even if a parent or child edge would reach into `domain`.\r\n\r\n* `connected(x)`: Same as `x::x`. Useful when `x` includes several commits.\r\n\r\n* `all()`: All visible commits and ancestors of commits explicitly mentioned.\r\n\r\n* `none()`: No commits. This function is rarely useful; it is provided for\r\n  completeness.\r\n\r\n* `change_id(prefix)`: Commits with the given change ID prefix. If the specified\r\n  change is divergent, this resolves to multiple commits. It is an error to use a\r\n  non-unique prefix. Unmatched prefix isn't an error.\r\n\r\n* `commit_id(prefix)`: Commits with the given commit ID prefix. It is an error\r\n  to use a non-unique prefix. Unmatched prefix isn't an error.\r\n\r\n* `bookmarks([pattern])`: All local bookmark targets. If `pattern` is specified,\r\n  this selects the bookmarks whose name match the given [string\r\n  pattern](#string-patterns). For example, `bookmarks(*push*)` would match the\r\n  bookmarks `push-123` and `repushed` but not the bookmark `main`. If a bookmark\r\n  is in a conflicted state, all its possible targets are included.\r\n\r\n* `remote_bookmarks([name_pattern], [[remote=]remote_pattern])`: All remote\r\n  bookmarks targets across all remotes. If just the `name_pattern` is specified,\r\n  the bookmarks whose names match the given [string pattern](#string-patterns)\r\n  across all remotes are selected. If both `name_pattern` and `remote_pattern`\r\n  are specified, the selection is further restricted to just the remotes whose\r\n  names match `remote_pattern`.\r\n\r\n  For example, `remote_bookmarks(*push*, *ri*)` would match the bookmarks\r\n  `push-123@origin` and `repushed@private` but not `push-123@upstream` or\r\n  `main@origin` or `main@upstream`. If a bookmark is in a conflicted state, all\r\n  its possible targets are included.\r\n\r\n  Git-tracking bookmarks are excluded by default. Use `remote=\"git\"` or\r\n  `remote=\"*\"` to select bookmarks including `@git` ones.\r\n\r\n* `tracked_remote_bookmarks([name_pattern], [[remote=]remote_pattern])`: All\r\n  targets of tracked remote bookmarks. Supports the same optional arguments as\r\n  `remote_bookmarks()`.\r\n\r\n* `untracked_remote_bookmarks([name_pattern], [[remote=]remote_pattern])`: All\r\n  targets of untracked remote bookmarks. Supports the same optional arguments as\r\n  `remote_bookmarks()`.\r\n\r\n* `tags([pattern])`: All tag targets. If `pattern` is specified, this selects\r\n  the tags whose name match the given [string pattern](#string-patterns). For\r\n  example, `tags(*v1*)` would match the tags `v123` and `rev1` but not the tag\r\n  `v2`. If a tag is in a conflicted state, all its possible targets are\r\n  included.\r\n\r\n* `remote_tags([name_pattern], [[remote=]remote_pattern])`: All remote tags\r\n  targets across all remotes. See `remote_bookmarks()` for arguments.\r\n\r\n* `visible_heads()`: All visible heads (same as `heads(all())` if no hidden\r\n  revisions are mentioned).\r\n\r\n* `root()`: The virtual commit that is the oldest ancestor of all other commits.\r\n\r\n* `heads(x)`: Commits in `x` that are not ancestors of other commits in `x`.\r\n  Equivalent to `x ~ ::x-`. Note that this is different from\r\n  [Mercurial's](https://repo.mercurial-scm.org/hg/help/revsets) `heads(x)`\r\n  function, which is equivalent to `x ~ x-`.\r\n\r\n* `roots(x)`: Commits in `x` that are not descendants of other commits in `x`.\r\n  Equivalent to `x ~ x+::`. Note that this is different from\r\n  [Mercurial's](https://repo.mercurial-scm.org/hg/help/revsets) `roots(x)`\r\n  function, which is equivalent to `x ~ x+`.\r\n\r\n* `latest(x, [count])`: Latest `count` commits in `x`, based on committer\r\n  timestamp. The default `count` is 1.\r\n\r\n* `fork_point(x)`: The fork point of all commits in `x`. The fork point is the\r\n  common ancestor(s) of all commits in `x` which do not have any descendants\r\n  that are also common ancestors of all commits in `x`. It is equivalent to\r\n  the revset `heads(::x_1 & ::x_2 & ... & ::x_N)`, where `x_{1..N}` are commits\r\n  in `x`. If `x` resolves to a single commit, `fork_point(x)` resolves to `x`.\r\n\r\n* `bisect(x)`: Finds commits in the input set for which about half of the input\r\n  set are descendants. The current implementation deals somewhat poorly with\r\n  non-linear history.\r\n\r\n* `exactly(x, count)`: Evaluates `x`, and errors if it is not of exactly size\r\n  `count`. Otherwise, returns `x`. This is useful in particular with `count=1`\r\n  when you want to ensure that some revset expression has exactly one target.\r\n\r\n* `merges()`: Merge commits.\r\n\r\n* `description(pattern)`: Commits that have a description matching the given\r\n  [string pattern](#string-patterns).\r\n\r\n  A non-empty description is usually terminated with newline character. For\r\n  example, `description(\"\")` matches commits without description, and\r\n  `description(\"foo\\n\")` matches commits with description `\"foo\\n\"`.\r\n\r\n* `subject(pattern)`: Commits that have a subject matching the given [string\r\n  pattern](#string-patterns). A subject is the first line of the description\r\n  (without newline character.)\r\n\r\n* `author(pattern)`: Commits with the author's name or email matching the given\r\n  [string pattern](#string-patterns). Equivalent to `author_name(pattern) |\r\n  author_email(pattern)`.\r\n\r\n* `author_name(pattern)`: Commits with the author's name matching the given\r\n  [string pattern](#string-patterns).\r\n\r\n* `author_email(pattern)`: Commits with the author's email matching the given\r\n  [string pattern](#string-patterns).\r\n\r\n* `author_date(pattern)`: Commits with author dates matching the specified [date\r\n  pattern](#date-patterns).\r\n\r\n* `mine()`: Commits where the author's email matches the email of the current\r\n  user. Equivalent to `author_email(exact-i:<user-email>)`\r\n\r\n* `committer(pattern)`: Commits with the committer's name or email matching the\r\n  given [string pattern](#string-patterns). Equivalent to\r\n  `committer_name(pattern) | committer_email(pattern)`.\r\n\r\n* `committer_name(pattern)`: Commits with the committer's name matching the\r\n  given [string pattern](#string-patterns).\r\n\r\n* `committer_email(pattern)`: Commits with the committer's email matching the\r\n  given [string pattern](#string-patterns).\r\n\r\n* `committer_date(pattern)`: Commits with committer dates matching the specified\r\n  [date pattern](#date-patterns).\r\n\r\n* `signed()`: Commits that are cryptographically signed.\r\n\r\n* `empty()`: Commits modifying no files. This also includes `merges()` without\r\n  user modifications and `root()`.\r\n\r\n* `files(expression)`: Commits modifying paths matching the given [fileset\r\n  expression](filesets.md).\r\n\r\n  Paths are relative to the directory `jj` was invoked from. A directory name\r\n  will match all files in that directory and its subdirectories.\r\n\r\n  For example, `files(foo)` will match files `foo`, `foo/bar`, `foo/bar/baz`.\r\n  It will *not* match `foobar` or `bar/foo`.\r\n\r\n  Some file patterns might need quoting because the `expression` must also be\r\n  parsable as a revset. For example, `.` has to be quoted in `files(\".\")`.\r\n\r\n* `diff_lines(text, [files])`: Commits containing diffs matching the given\r\n  `text` pattern line by line.\r\n\r\n  The search paths can be narrowed by the `files` expression. All modified files\r\n  are scanned by default, but it is likely to change in future version to\r\n  respect the command line path arguments.\r\n\r\n  For example, `diff_lines(\"*TODO*\", \"src\")` will search revisions where \"TODO\"\r\n  is added to or removed from files under \"src\".\r\n\r\n* `conflicts()`: Commits that have files in a conflicted state.\r\n\r\n* `divergent()`: Commits that are [divergent](glossary.md#divergent-change).\r\n\r\n* `present(x)`: Same as `x`, but evaluated to `none()` if any of the commits\r\n  in `x` doesn't exist (e.g. is an unknown bookmark name.)\r\n\r\n* `coalesce(revsets...)`: Commits in the first revset in the list of `revsets`\r\n  which does not evaluate to `none()`. If all revsets evaluate to `none()`, then\r\n  the result of `coalesce` will also be `none()`.\r\n\r\n* `working_copies()`: The working copy commits across all the workspaces.\r\n\r\n* `at_operation(op, x)`: Evaluates `x` at the specified [operation][]. For\r\n  example, `at_operation(@-, visible_heads())` will return all heads which were\r\n  visible at the previous operation.\r\n\r\n  Since `at_operation(op, x)` brings all commits that were visible at the\r\n  operation to the search space, `at_operation(op, x) | all()` is equivalent to\r\n  `at_operation(op, x) | ::(at_operation(op, x | visible_heads()) |\r\n  visible_heads())`.\r\n\r\n[operation]: glossary.md#operation\r\n\r\n??? examples\r\n\r\n    Given this history:\r\n    ```\r\n    o E\r\n    |\r\n    | o D\r\n    |/|\r\n    | o C\r\n    | |\r\n    o | B\r\n    |/\r\n    o A\r\n    |\r\n    o root()\r\n    ```\r\n\r\n    **function** `reachable()`\r\n\r\n    * `reachable(E, A..)` ⇒ `{E,D,C,B}`\r\n    * `reachable(D, A..)` ⇒ `{E,D,C,B}`\r\n    * `reachable(C, A..)` ⇒ `{E,D,C,B}`\r\n    * `reachable(B, A..)` ⇒ `{E,D,C,B}`\r\n    * `reachable(A, A..)` ⇒ `{}` (empty set)\r\n\r\n    **function** `connected()`\r\n\r\n    * `connected(E|A)` ⇒ `roots(E|B)` ⇒ `{B}`\r\n    * `roots(E|A)` ⇒ `{A}`\r\n    * `roots(A)` ⇒ `{A}`\r\n\r\n    **function** `fork_point()`\r\n\r\n    * `fork_point(E|D)` ⇒ `{B}`\r\n    * `fork_point(E|C)` ⇒ `{A}`\r\n    * `fork_point(E|B)` ⇒ `{B}`\r\n    * `fork_point(E|A)` ⇒ `{A}`\r\n    * `fork_point(D|C)` ⇒ `{C}`\r\n    * `fork_point(D|B)` ⇒ `{B}`\r\n    * `fork_point(B|C)` ⇒ `{A}`\r\n    * `fork_point(A)` ⇒ `{A}`\r\n    * `fork_point(none())` ⇒ `{}`\r\n\r\n## String patterns\r\n\r\nFunctions that perform string matching support the following pattern syntax (the\r\nquotes are optional).\r\n\r\nBy default, `\"string\"` is parsed as a `glob:` pattern.\r\n\r\n* `exact:\"string\"`: Matches strings exactly equal to `string`.\r\n* `glob:\"pattern\"`: Matches strings with Unix-style shell [wildcard\r\n  `pattern`](https://docs.rs/globset/latest/globset/#syntax).\r\n* `regex:\"pattern\"`: Matches substrings with [regular\r\n  expression `pattern`](https://docs.rs/regex/latest/regex/#syntax).\r\n* `substring:\"string\"`: Matches strings that contain `string`.\r\n\r\nYou can append `-i` after the kind to match case‐insensitively (e.g.\r\n`glob-i:\"fix*jpeg*\"`).\r\n\r\nString patterns can be combined by using logical operators\r\n(e.g. `bookmarks(~glob:\"ci/*\")`):\r\n\r\n* `~x`: Matches everything but `x`.\r\n* `x & y`: Matches both `x` and `y`.\r\n* `x ~ y`: Matches `x` but not `y`.\r\n* `x | y`: Matches either `x` or `y` (or both).\r\n\r\n## Date patterns\r\n\r\nFunctions that perform date matching support the following pattern syntax:\r\n\r\n* `after:\"string\"`: Matches dates exactly at or after the given date.\r\n* `before:\"string\"`: Matches dates before, but not including, the given date.\r\n\r\nDate strings can be specified in several forms, including:\r\n\r\n* 2024-02-01\r\n* 2024-02-01T12:00:00\r\n* 2024-02-01T12:00:00-08:00\r\n* 2024-02-01 12:00:00\r\n* 2 days ago\r\n* 5 minutes ago\r\n* yesterday\r\n* yesterday 5pm\r\n* yesterday 10:30\r\n* yesterday 15:30\r\n\r\n## Aliases\r\n\r\nNew symbols and functions can be defined in the config file, by using any\r\ncombination of the predefined symbols/functions and other aliases.\r\n\r\nAlias functions can be overloaded by the number of parameters. However, builtin\r\nfunction will be shadowed by name, and can't co-exist with aliases.\r\n\r\nFor example:\r\n\r\n```toml\r\n[revset-aliases]\r\n'HEAD' = '@-'\r\n'user()' = 'user(\"me@example.org\")'\r\n'user(x)' = 'author(x) | committer(x)'\r\n```\r\n\r\n### Built-in Aliases\r\n\r\nThe following aliases are built-in and used for certain operations. These functions\r\nare defined as aliases in order to allow you to overwrite them as needed.\r\nSee [revsets.toml](https://github.com/jj-vcs/jj/blob/main/cli/src/config/revsets.toml)\r\nfor a comprehensive list.\r\n\r\n* `trunk()`: Resolves to the head commit for the default bookmark of the default\r\n  remote, or the remote named `upstream` or `origin`. This is set at the\r\n  repository level upon initialization of a Jujutsu repository.\r\n\r\n  If the default bookmark cannot be resolved during initialization, the default\r\n  global configuration tries the bookmarks `main`, `master`, and `trunk` on the\r\n  `upstream` and `origin` remotes. If more than one potential trunk commit\r\n  exists, the newest one is chosen.
